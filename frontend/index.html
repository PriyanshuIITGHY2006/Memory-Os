Architecture
Scalable Persistent Memory for LLMs
How MemoryOS maintains coherence across thousands of turns without degrading token efficiency or recall accuracy.

The Stationarity Problem

Naive Approach — O(N) token cost
Concatenating full conversation history grows proportionally with turn count. At 4000 turns this saturates the context window and explodes API cost. Recall degrades as distant memories are truncated.
MemoryOS — O(1) token cost
A relevance-scored budget system selects the top-K memory nodes by score and fills a fixed character budget. Token cost per turn is constant regardless of conversation length.

Relevance Scoring

Memory Relevance Score
$$\text{score}(m_i \mid Q) = \alpha \cdot \underbrace{\frac{|Q \cap W_i|}{|Q|}}_{\text{keyword}} + \beta \cdot \underbrace{e^{-\lambda \Delta t}}_{\text{recency}} + \gamma \cdot \underbrace{\frac{\ln(1+f_i)}{\ln(101)}}_{\text{frequency}}$$
where $\alpha = 0.50$, $\beta = 0.30$, $\gamma = 0.20$, $\lambda = 0.001$, $\Delta t = t_{\text{now}} - t_{\text{last access}}$

Greedy Context Selection

Fixed-Budget Selection
$$S^* = \arg\max_{S \subseteq \mathcal{M}} \sum_{m_i \in S} \text{score}(m_i \mid Q) \quad \text{subject to} \quad \sum_{m_i \in S} |m_i| \leq B$$
Solved greedily by sorting $\mathcal{M}$ by score descending and filling budget $B$ (characters). Budget $B$ is fixed at compile time — independent of $|\mathcal{M}|$.
Token Complexity Comparison
$$\text{tokens}_{\text{naive}}(N) = \Theta(N) \qquad \text{tokens}_{\text{MemoryOS}}(N) = \Theta(1)$$
$N$ = number of prior conversation turns. MemoryOS maintains constant token usage regardless of $N$.

Entity Resolution

Duplicate Confidence Score
$$c(e_a, e_b) = \mathbf{1}[\text{rel}_a = \text{rel}_b] \cdot 0.40 + \mathbf{1}[e_a \subset e_b] \cdot 0.50 + \text{lev}(e_a, e_b) \cdot 0.40 + \mathbf{1}[\text{possessive}(e_a)] \cdot 0.60$$
where $\text{lev}(a,b)$ is the normalised Levenshtein similarity and $\text{possessive}(e)$ detects "X's Y" patterns. Pairs with $c \geq 0.40$ are flagged for merge.

Request Processing Pipeline

1
Turn Increment
Atomic Neo4j counter write. Provides monotonic turn ID used in recency decay $e^{-\lambda \Delta t}$.
2
Proactive Semantic Search
ChromaDB cosine similarity search over last 3 buffer turns. Top-4 episodic memories retrieved before context assembly.
3
Relevance-Scored Context Assembly
ContextBuilder scores all Neo4j nodes against query tokens. Greedy fill to fixed budget. Compact pipe-delimited serialization (~88 tokens typical).
4
LLM Inference — Groq API
9-tool schema injected. Agentic loop up to 4 rounds for multi-tool turns. Parallel tool calls enabled. llama-3.3-70b-versatile.
5
Tool Dispatch
Graph writes (Neo4j): update_entity, link_entities, log_event, merge_entities. Archive reads: archival_memory_search.
6
Background Episode Save
Full exchange saved to ChromaDB in daemon thread. Non-blocking — user receives response before archive write completes.

Storage Architecture

Neo4j AuraDB — Structured Memory
Graph database storing the User World Model as a labeled property graph. Node labels: User, Entity, Event, Knowledge, Preference. Edge types: KNOWS, RELATED_TO, INVOLVED_IN, HAS_PREFERENCE. Enables multi-hop traversal and structural deduplication via APOC-free Cypher merge.
ChromaDB — Episodic Memory
Local persistent vector store for full conversation episodes. Each exchange stored as a document with turn metadata. Cosine similarity search returns semantically relevant past episodes that may not be present in the graph's structured form. Complements Neo4j with unstructured long-term recall.
